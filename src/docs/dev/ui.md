# Reusable UI Components

## Purpose

This project includes a reusable UI library that is shared across all projects.

These components are considered **core infrastructure** and should be reused whenever possible.

**Do not recreate these components.**

Always import and use the existing implementation.

---

# Agent Rules

When generating UI:

- Reuse existing UI components before creating new ones.
- Do not duplicate functionality that already exists.
- Extend existing components through props, variants, or composition.
- Only create a new component when no existing component satisfies the requirement.

Priority:

```text
Existing Component
        ↓
Extend Existing Component
        ↓
Create New Component (Last Resort)
```

---

# Button System

Location

```text
ui/buttons/Button.tsx
```

Purpose

Centralized button component used throughout the application.

The component already supports project-wide styling and should be considered the single source of truth for buttons.

## Agent Rules

Always use:

```tsx
<Button />
```

Never generate:

```tsx
<button />
```

unless explicitly requested.

If a new design is required:

- add a new variant
- add a new size
- extend existing props

Do **not** create another Button component.

Examples

```tsx
<Button variant="primary" />

<Button variant="secondary" />

<Button variant="ghost" />

<Button variant="outline" />
```

---

# Input System

Location

```text
ui/inputs/InputFactory.tsx
```

Purpose

Centralized factory responsible for rendering all input types.

The factory internally selects the correct input component using a switch-case architecture.

## Agent Rules

Never build individual input components directly.

Always use:

```tsx
<InputFactory />
```

Supported examples include:

- text
- email
- password
- number
- textarea
- select
- checkbox
- radio
- switch
- date
- search

If a new input is needed:

- extend InputFactory
- add a new input type
- keep a consistent API

Do not create independent input implementations.

# Link System

Location

```text
ui/links/Link.tsx
```

Purpose

Unified navigation component for internal and external links.

The component contains the project's default styling and navigation behavior.

## Agent Rules

Always use:

```tsx
<Link />
```

Do not use:

```tsx
<a />;

NextLink;
```

directly unless specifically required.

Styling belongs inside the Link component.

# SVG Icon System

Location

```text
ui/svg/IconSVG.tsx
```

Purpose

Renders custom SVG assets used throughout the application.

## Agent Rules

Always use:

```tsx
<IconSVG />
```

Do not inline SVG markup into components.

New SVG assets should be added to the SVG system rather than embedded in pages.

---

# Lucide Icon System

Location

```text
ui/images/LucideIcon.tsx
```

Purpose

Server-only wrapper around `lucide-react`.

Provides a centralized API for rendering Lucide icons.

## Agent Rules

Always use:

```tsx
<LucideIcon iconName="Home" />
```

Never import icons directly:

```tsx
import { Home } from "lucide-react";
```

unless explicitly requested.

The wrapper ensures consistent usage and simplifies maintenance.

# Shadcn UI

Location

```text
ui/shadcn/
```

Purpose

Contains all components generated by Shadcn UI.

Examples

```text
Button
Dialog
DropdownMenu
Popover
Card
Sheet
Tabs
Accordion
Avatar
Badge
```

## Agent Rules

Do not regenerate Shadcn components.

If a required component already exists inside:

```text
ui/shadcn
```

reuse it.

If a component is missing:

1. Install it using the Shadcn CLI.
2. Place it inside:

```text
ui/shadcn/
```

3. Follow the existing project structure.

Do not manually recreate Shadcn components.

# Styling Responsibility

Global styles define:

- design tokens
- typography
- spacing
- colors
- motion
- accessibility

Component appearance belongs inside the reusable UI components.

Examples

```text
Button.tsx
InputFactory.tsx
Link.tsx
IconSVG.tsx
LucideIcon.tsx
```

Pages and feature components should focus only on:

- layout
- composition
- business logic

They should not redefine component styling.

---

# AI Decision Tree

Before generating UI, always follow this order:

```text
Need Button?
    → Use ui/buttons/Button.tsx

Need Input?
    → Use ui/inputs/InputFactory.tsx

Need Link?
    → Use ui/links/Link.tsx

Need Custom SVG?
    → Use ui/svg/IconSVG.tsx

Need Lucide Icon?
    → Use ui/images/LucideIcon.tsx

Need Shadcn Component?
    → Check ui/shadcn/

Need New Component?
    → Create only if no reusable solution exists.
```

# Core Principle

These reusable UI components are part of the project's design system.

They should be treated as stable infrastructure.

Agents should prefer extending existing components over creating new ones, ensuring visual consistency, maintainability, and minimal code duplication across every project.
